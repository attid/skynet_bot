# 2026-05-08: Wrong emoji captcha message

## Контекст
- При неверном ответе emoji-капчи бот удаляет inline-кнопки, пользователь остается restricted, но сообщение выглядит как капча без кнопки.
- Нужно оставить one-shot поведение, явно показать пользователю причину и контакт администрации, но убрать строки самой капчи, чтобы не оставлять невыполнимую инструкцию.

## План изменений
1. [x] Добавить regression test для неверного ответа emoji-капчи в `tests/routers/test_welcome.py`.
2. [x] Изменить ветку неверного `EmojiCaptchaCallbackData` в `routers/welcome.py`: редактировать исходный текст, удаляя строки с captcha emoji, добавляя отказ и контакт `@mtl_helper_bot`, с удалением клавиатуры.
3. [x] Добавить явный moderation log для неверного ответа.
4. [x] Обновить/добавить тесты.
5. [x] Обновить docs/ (если затронуты контракты/поведение).
6. [x] Проверка: `uv run pytest -q tests/routers/test_welcome.py`.

## Риски и открытые вопросы
- Telegram может отказать в `edit_text` для недоступного/неизмененного сообщения; нужен fallback на удаление `reply_markup`.
- Исходный welcome-текст отправляется в HTML parse mode, поэтому при редактировании нужно сохранить HTML-совместимое поведение.

## Верификация
- Неверный emoji callback должен редактировать сообщение, вырезать строки с captcha emoji, убирать клавиатуру и не восстанавливать права.
- Успешная emoji-капча должна продолжить восстанавливать права.
- Ожидаемый лог: `moderation_artifact action=captcha_wrong_answer ... source=captcha_v2`.
